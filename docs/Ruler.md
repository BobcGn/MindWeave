# RULER.md

## 1. 文档目的
本文件用于约束 Codex 在本仓库中的实现方式、代码风格、架构边界与交付格式。

Codex 在执行任何任务时，必须优先遵守本文件中的规则。

---

## 2. 项目定位
本项目是一个 **半离线、隐私优先、支持多设备同步的个人日记应用**。

核心功能包括：
- 日记编写
- 日程管理
- AI 对话
- 本地优先存储
- 多设备同步
- 云端增强能力（同步、AI、备份）

本项目不是纯在线应用。
本地数据库始终是每台设备上的事实数据源。

---

## 3. 技术栈约束

### 3.1 客户端
- 使用 Kotlin Multiplatform（KMP）作为共享核心
- Android 使用原生 Android 技术栈
- iOS 使用原生 iOS 技术栈
- 本地数据库使用 SQLite
- KMP 数据库层优先使用 SQLDelight
- 网络层优先使用 Ktor Client
- 序列化优先使用 kotlinx.serialization
- 并发与异步优先使用 Coroutines + Flow

### 3.2 服务端
- 使用 Kotlin 技术栈
- 优先使用 Ktor 作为服务端框架
- 数据库优先使用 PostgreSQL
- 服务端不负责替代本地数据库
- 服务端主要负责同步、AI 编排、备份、设备管理

### 3.3 AI
- AI 能力必须通过抽象层接入
- 不允许在 UI 或业务层直接耦合具体 AI SDK
- Koog 只能作为某个实现，不应成为唯一耦合点
- 设计上必须允许未来切换为其他云模型或本地模型

---

## 4. 架构约束

### 4.1 总体原则
必须采用 **本地优先（offline-first）** 架构。

基本原则：
1. 用户写入数据时，必须优先写入本地数据库
2. 本地写入不能依赖网络成功
3. 同步必须在数据库之上实现，而不是依赖数据库自身复制能力
4. 服务端是全局协调中心，不是用户编辑体验的直接入口
5. 多设备同步必须通过服务端中转，不允许直接设备对设备同步

### 4.2 模块边界
推荐模块：
- `shared`
- `androidApp`
- `iosApp`
- `server`

其中：
- `shared`：共享领域模型、仓储接口、数据库 schema、同步逻辑、AI 抽象
- `androidApp`：Android UI 与平台能力
- `iosApp`：iOS UI 与平台能力
- `server`：同步、认证、AI 编排、备份

### 4.3 shared 模块推荐包结构
- `domain`
- `database`
- `data`
- `repository`
- `sync`
- `ai`
- `util`

## 5. 数据建模规则

### 5.1 所有可同步实体必须具备以下字段
每一个可同步实体都必须包含以下字段：

- `id`
- `userId`
- `createdAt`
- `updatedAt`
- `deletedAt`
- `version`
- `lastModifiedByDeviceId`

说明：
- `id`：全局唯一标识，优先使用 UUID 或 ULID
- `userId`：归属用户
- `createdAt`：创建时间
- `updatedAt`：最后修改时间
- `deletedAt`：软删除时间，可为空
- `version`：版本号，用于同步与冲突处理
- `lastModifiedByDeviceId`：最后修改该记录的设备标识

### 5.2 删除策略
所有可同步实体必须采用 **软删除**，禁止直接物理删除。

### 5.3 主键规则
禁止依赖本地自增整数作为跨端同步主键。
跨端同步主键必须是字符串型全局唯一 ID。

---

## 6. 同步规则

### 6.1 同步总原则
同步必须采用 **服务端中转的增量同步** 方案。

基本流程：
1. 本地修改先写 SQLite
2. 本地记录待同步变更
3. 客户端向服务端推送本地变更
4. 客户端从服务端拉取其他设备产生的变更
5. 将远端变更应用到本地 SQLite

### 6.2 同步方式
必须支持：
- 增量同步
- 软删除同步
- 冲突检测
- 幂等处理
- 失败重试

### 6.3 初版冲突策略
初版统一采用简单策略：

- 日记：`Last Write Wins`
- 日程：`Last Write Wins`
- 标签：并集合并
- 聊天消息：尽量设计为追加型，避免编辑冲突

### 6.4 本地同步辅助结构
客户端必须预留以下同步能力：
- outbox / 待同步队列
- sync metadata（如最后同步游标）
- 冲突记录（可选但推荐）
- 重试机制

---

## 7. 数据库规则

### 7.1 客户端数据库
客户端必须使用 SQLite。
KMP 层优先通过 SQLDelight 统一管理 schema 与查询。

### 7.2 服务端数据库
服务端数据库优先使用 PostgreSQL。
服务端不建议使用 SQLite 作为中心同步数据库。

### 7.3 Migration
只要 schema 发生变化，必须补充 migration。
禁止无 migration 的 schema 变更。

### 7.4 索引要求
必须为以下字段设计合理索引：
- `userId`
- `updatedAt`
- `deletedAt`
- 外键
- 日程时间范围查询相关字段

---

## 8. AI 接入规则

### 8.1 必须抽象
必须定义统一 AI 接口，例如：
- `AiAgent`
- `AiRequest`
- `AiResponse`
- `ChatContext`

### 8.2 上下文来源
AI 上下文组装应支持以下来源：
- 最近日记
- 即将到来的日程
- 最近对话历史
- 用户偏好（后续可扩展）

### 8.3 隐私原则
AI 数据流必须清晰、显式。
不得在不透明位置自动上传用户隐私数据。
架构上必须允许未来关闭云端 AI 或切换本地模型。

---

## 9. 代码实现规则

### 9.1 通用原则
- 优先写清晰代码，不要过度抽象
- 优先可测试性
- 优先可维护性
- 优先小步提交，不要一次性大改
- 不允许为了“看起来高级”引入不必要复杂度

### 9.2 平台代码
平台专属代码必须控制在必要范围内。
能在 shared 实现的逻辑，不要下沉到平台层。

### 9.3 命名
命名要直接、语义明确。
避免模糊缩写。

### 9.4 注释与文档
所有公开接口和关键模块应具备基础说明。
复杂同步逻辑必须补充注释或文档。

---

## 10. 测试规则

必须为核心逻辑编写测试，优先覆盖：
- 仓储层
- SQLDelight 查询行为
- 同步逻辑
- 冲突处理
- AI 上下文组装

至少需要验证：
- 新增
- 更新
- 软删除
- 增量同步
- 重复同步幂等性
- 基本冲突场景

---

## 11. 禁止事项

Codex 不应做以下事情：

- 不要直接把所有逻辑塞进 UI 层
- 不要让服务端替代本地数据库
- 不要在没有抽象层的情况下直接耦合 AI SDK
- 不要使用数据库文件级同步替代实体级同步
- 不要默认使用复杂 CRDT 方案
- 不要在未说明的情况下引入重量级框架
- 不要破坏已有模块编译
- 不要修改 schema 却不写 migration
- 不要物理删除可同步数据

---

## 12. 每次任务完成后的输出格式

Codex 每次完成任务后，必须输出以下内容：

1. 本次修改了什么
2. 涉及哪些文件
3. 是否包含 schema 或 API 变化
4. 是否新增 migration
5. 当前假设与取舍
6. 下一步最推荐做什么

---

## 13. 实现优先级
如果没有额外说明，默认按以下顺序推进：

1. 架构与模块边界
2. 本地数据库 schema
3. 仓储层
4. 同步骨架
5. 服务端同步 MVP
6. AI 抽象层
7. UI 接入
8. 高级功能（搜索、总结、备份、推送等）

---

## 14. 最终目标
该项目的目标不是做一个“在线 CRUD 应用”，而是做一个：

**本地优先、跨设备同步、可扩展 AI 能力、隐私友好的个人日记系统。**
