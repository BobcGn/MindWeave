# RULER.md

## 1. 文档目的
本文件用于约束 Codex 在本仓库中的实现方式、代码风格、架构边界与交付格式。

Codex 在执行任何任务时，必须优先遵守本文件中的规则。

---

## 2. 项目定位
本项目是一个 **半离线、隐私优先、支持多设备同步的个人日记应用**。

核心功能包括：
- 日记编写
- 日程管理
- AI 对话
- 本地优先存储
- 多设备同步
- 云端增强能力（同步、AI、备份）

本项目不是纯在线应用。
本地数据库始终是每台设备上的事实数据源。

---

## 3. 技术栈约束

### 3.1 客户端
- 使用 Kotlin Multiplatform（KMP）作为共享核心
- Android 使用原生 Android 技术栈
- iOS 使用原生 iOS 技术栈
- 本地数据库使用 SQLite
- KMP 数据库层优先使用 SQLDelight
- 网络层优先使用 Ktor Client
- 序列化优先使用 kotlinx.serialization
- 并发与异步优先使用 Coroutines + Flow

### 3.2 服务端
- 使用 Kotlin 技术栈
- 优先使用 Ktor 作为服务端框架
- 数据库优先使用 PostgreSQL
- 服务端不负责替代本地数据库
- 服务端主要负责同步、AI 编排、备份、设备管理

### 3.3 AI
- AI 能力必须通过抽象层接入
- 不允许在 UI 或业务层直接耦合具体 AI SDK
- Koog 只能作为某个实现，不应成为唯一耦合点
- 设计上必须允许未来切换为其他云模型或本地模型

---

## 4. 架构约束

### 4.1 总体原则
必须采用 **本地优先（offline-first）** 架构。

基本原则：
1. 用户写入数据时，必须优先写入本地数据库
2. 本地写入不能依赖网络成功
3. 同步必须在数据库之上实现，而不是依赖数据库自身复制能力
4. 服务端是全局协调中心，不是用户编辑体验的直接入口
5. 多设备同步必须通过服务端中转，不允许直接设备对设备同步

### 4.2 模块边界
推荐模块：
- `shared`
- `androidApp`
- `iosApp`
- `server`

其中：
- `shared`：共享领域模型、仓储接口、数据库 schema、同步逻辑、AI 抽象
- `androidApp`：Android UI 与平台能力
- `iosApp`：iOS UI 与平台能力
- `server`：同步、认证、AI 编排、备份

### 4.3 shared 模块推荐包结构
- `domain`
- `database`
- `data`
- `repository`
- `sync`
- `ai`
- `util`

### 4.4 HarmonyOS / 鸿蒙跨端实践约束

#### 4.4.1 总体原则

鸿蒙端接入必须遵循与 Android / iOS 一致的整体架构原则：

* 共享业务逻辑优先放在 shared
* 鸿蒙端仅承载 UI、平台能力适配与必要的桥接层
* 本地数据仍以设备侧 SQLite 为事实数据源
* 同步、AI、备份等云端能力仍通过既有服务端体系完成
* 不允许为了支持鸿蒙而破坏现有 Android / iOS / server 的模块边界

鸿蒙接入的目标是：
在不重写核心业务逻辑的前提下，将 KMP 的共享能力扩展到 HarmonyOS 平台。

⸻

#### 4.4.2 模块建议

在现有模块基础上，允许增加以下模块之一：

推荐方式：

* harmonyApp

可选方式：

* harmonyApp
* shared
* server

其中：

* shared：继续承载领域模型、仓储接口、数据库 schema、同步逻辑、AI 抽象、通用用例
* harmonyApp：鸿蒙 UI、平台生命周期、系统能力接入、共享层桥接
* server：不因鸿蒙接入而新增平台特化逻辑，除非确有设备能力差异需要建模

如果鸿蒙平台需要额外的平台实现，允许在 shared 中增加：

* harmonyMain

但必须保持其职责为：

* 平台 API 适配
* 文件系统/日志/时间/网络等平台能力 actual 实现
* 必要的线程或桥接封装

不得将鸿蒙页面逻辑、路由逻辑、ArkUI 组件实现放入 shared。

⸻

#### 4.4.3 UI 策略

鸿蒙端必须采用 平台原生 UI 思路。

基本要求：

* shared 不负责鸿蒙 UI 渲染
* 鸿蒙 UI 应由 ArkUI / ArkTS 或鸿蒙原生能力承载
* shared 负责提供状态、领域逻辑、数据访问接口、同步能力、AI 抽象
* harmonyApp 负责页面、交互、生命周期、导航、权限申请、平台能力编排

禁止事项：

* 不要在初版中为了“全平台统一 UI”引入额外高复杂度 UI 跨端方案
* 不要为了鸿蒙端接入而重构 Android / iOS UI 层为非原生方案
* 不要把平台页面状态管理与共享领域状态强耦合到不可测试的桥接代码中

推荐模式：

* shared 输出 ViewModel 所需状态模型或 use case
* 鸿蒙端自行订阅、转换并渲染为平台 UI

⸻

#### 4.4.4 共享层接入策略

鸿蒙接入必须优先复用以下共享能力：

* 领域模型
* Repository 接口
* SQLDelight schema 与查询定义
* 同步逻辑
* AI 抽象层
* 序列化模型
* 网络协议定义
* 通用 use case

鸿蒙端不应重复实现以下内容：

* 实体定义
* 同步协议处理
* 冲突处理规则
* AI 请求/响应抽象
* 数据序列化规则

如果平台差异导致必须新增实现，应优先通过 expect / actual 机制扩展，不要复制整套业务逻辑。

⸻

#### 4.4.5 平台桥接策略

KMP 跨到鸿蒙时，允许采用桥接方式接入，但必须保持桥接层薄、职责明确。

可接受的桥接职责包括：

* 共享层导出接口的绑定
* 基础类型转换
* 异步回调/事件派发适配
* 平台日志、文件、时间、网络能力封装
* Flow / suspend 到鸿蒙可调用形式的适配

桥接层禁止承担：

* 业务规则实现
* 同步冲突处理
* AI 上下文拼装
* Repository 核心逻辑
* 数据库 schema 管理

要求：

* 桥接代码必须是薄层
* 桥接接口命名必须清晰
* 桥接层输入输出必须尽量稳定，避免把内部实现细节暴露给鸿蒙 UI 层

⸻

#### 4.4.6 推荐落地路线

在没有额外说明的情况下，鸿蒙接入默认按以下路线推进：

1. 先保证 shared 逻辑对平台无假设
2. 补齐 expect / actual 的平台能力抽象
3. 让鸿蒙端优先接入只读能力
4. 再接入本地写入与 SQLite
5. 再接入同步流程
6. 最后接入 AI 与复杂交互

初版推荐优先打通：

* 日记列表读取
* 日记详情读取
* 新建/编辑日记写本地库
* 本地同步队列生成
* 与服务端的增量同步闭环

不建议初版优先做：

* 全量高级编辑器能力
* 复杂富文本
* 高级推送/后台唤醒
* 大量平台特化动画或复杂 UI 容器

⸻

#### 4.4.7 数据库与存储要求

鸿蒙端依然必须遵循本项目的本地优先规则：

* 用户写入必须先落本地
* 本地数据库必须是鸿蒙设备上的事实数据源
* 不允许直接把鸿蒙端做成纯接口壳
* 不允许写操作依赖网络返回后才落本地

鸿蒙端如接入 SQLite：

* 必须继续复用 shared 中定义的 schema 与查询
* 优先保证与 Android / iOS 一致的数据建模
* 不允许针对鸿蒙单独分叉一套实体结构，除非有明确文档说明

如果鸿蒙平台在 SQLite 驱动接入方式上存在差异：

* 允许新增平台数据库 driver 适配
* 但不允许修改既有同步实体字段约束
* 不允许绕开 SQLDelight schema 单独维护一套表结构

⸻

#### 4.4.8 并发与异步约束

鸿蒙接入必须继续遵循：

* Coroutines + Flow 优先
* 共享层异步模型不能因鸿蒙接入而退化为回调地狱
* 平台侧只做必要的协程/事件循环适配

要求：

* suspend 方法应在桥接层封装为鸿蒙可消费接口
* Flow 应在桥接层转为平台可订阅状态流或事件流
* 主线程切换、生命周期取消、资源释放必须明确处理

禁止：

* 在平台层随意创建不可控后台任务
* 在桥接层吞掉异常
* 在 UI 层直接拼装复杂协程业务流程

⸻

#### 4.4.9 网络与 AI 约束

鸿蒙端必须尽量复用 shared 中已有：

* Ktor Client 抽象
* 序列化模型
* AI 接口定义
* 服务端协议

鸿蒙平台适配仅允许处理：

* HTTP engine 接入差异
* TLS / 证书 / 网络权限差异
* 平台文件上传下载接口差异
* 平台任务调度差异

禁止：

* 在鸿蒙端单独实现一套 AI 接口
* 在鸿蒙 UI 代码中直接耦合具体 AI SDK
* 在鸿蒙端绕过既有服务端协议私自定义另一套同步 API

⸻

#### 4.4.10 测试要求

鸿蒙接入后，测试优先级仍应以共享层为主。

至少需要验证：

* 鸿蒙平台可正常调用共享用例
* 本地新增/更新/软删除流程可走通
* 同步队列生成逻辑在鸿蒙端可正常触发
* Flow / suspend 桥接后的基础状态更新可用
* 关键平台 actual 实现行为正确

要求：

* 核心业务测试仍优先写在 shared
* 鸿蒙模块测试重点关注桥接层、生命周期、平台能力适配
* 不要把本应在共享层验证的逻辑测试全部下沉到鸿蒙 UI 层

⸻

#### 4.4.11 禁止事项

鸿蒙接入时，额外禁止以下行为：

* 不要为鸿蒙单独复制一套领域层
* 不要为鸿蒙单独复制一套同步逻辑
* 不要让鸿蒙端直接依赖服务端作为唯一数据源
* 不要把桥接层写成新的业务层
* 不要为了跨鸿蒙而破坏现有 KMP shared 的纯净性
* 不要在没有明确收益的情况下引入高复杂度跨 UI 框架
* 不要在鸿蒙端直接绕过 Repository 访问底层实现
* 不要为了平台兼容临时删减同步字段、软删除、版本号等关键约束

⸻

#### 4.4.12 实现优先级

如果任务是“为现有项目增加鸿蒙支持”，默认按以下顺序推进：

1. 清理 shared 中不合理的平台耦合
2. 提炼/补齐 expect / actual 能力抽象
3. 增加 harmonyApp 模块骨架
4. 打通共享层调用链路
5. 打通 SQLite 与 Repository
6. 打通基础日记读写
7. 打通同步 MVP
8. 打通 AI 能力
9. 最后补齐体验优化与平台特性

## 5. 数据建模规则

### 5.1 所有可同步实体必须具备以下字段
每一个可同步实体都必须包含以下字段：

- `id`
- `userId`
- `createdAt`
- `updatedAt`
- `deletedAt`
- `version`
- `lastModifiedByDeviceId`

说明：
- `id`：全局唯一标识，优先使用 UUID 或 ULID
- `userId`：归属用户
- `createdAt`：创建时间
- `updatedAt`：最后修改时间
- `deletedAt`：软删除时间，可为空
- `version`：版本号，用于同步与冲突处理
- `lastModifiedByDeviceId`：最后修改该记录的设备标识

### 5.2 删除策略
所有可同步实体必须采用 **软删除**，禁止直接物理删除。

### 5.3 主键规则
禁止依赖本地自增整数作为跨端同步主键。
跨端同步主键必须是字符串型全局唯一 ID。

---

## 6. 同步规则

### 6.1 同步总原则
同步必须采用 **服务端中转的增量同步** 方案。

基本流程：
1. 本地修改先写 SQLite
2. 本地记录待同步变更
3. 客户端向服务端推送本地变更
4. 客户端从服务端拉取其他设备产生的变更
5. 将远端变更应用到本地 SQLite

### 6.2 同步方式
必须支持：
- 增量同步
- 软删除同步
- 冲突检测
- 幂等处理
- 失败重试

### 6.3 初版冲突策略
初版统一采用简单策略：

- 日记：`Last Write Wins`
- 日程：`Last Write Wins`
- 标签：并集合并
- 聊天消息：尽量设计为追加型，避免编辑冲突

### 6.4 本地同步辅助结构
客户端必须预留以下同步能力：
- outbox / 待同步队列
- sync metadata（如最后同步游标）
- 冲突记录（可选但推荐）
- 重试机制

---

## 7. 数据库规则

### 7.1 客户端数据库
客户端必须使用 SQLite。
KMP 层优先通过 SQLDelight 统一管理 schema 与查询。

### 7.2 服务端数据库
服务端数据库优先使用 PostgreSQL。
服务端不建议使用 SQLite 作为中心同步数据库。

### 7.3 Migration
只要 schema 发生变化，必须补充 migration。
禁止无 migration 的 schema 变更。

### 7.4 索引要求
必须为以下字段设计合理索引：
- `userId`
- `updatedAt`
- `deletedAt`
- 外键
- 日程时间范围查询相关字段

---

## 8. AI 接入规则

### 8.1 必须抽象
必须定义统一 AI 接口，例如：
- `AiAgent`
- `AiRequest`
- `AiResponse`
- `ChatContext`

### 8.2 上下文来源
AI 上下文组装应支持以下来源：
- 最近日记
- 即将到来的日程
- 最近对话历史
- 用户偏好（后续可扩展）

### 8.3 隐私原则
AI 数据流必须清晰、显式。
不得在不透明位置自动上传用户隐私数据。
架构上必须允许未来关闭云端 AI 或切换本地模型。

---

## 9. 代码实现规则

### 9.1 通用原则
- 优先写清晰代码，不要过度抽象
- 优先可测试性
- 优先可维护性
- 优先小步提交，不要一次性大改
- 不允许为了“看起来高级”引入不必要复杂度

### 9.2 平台代码
平台专属代码必须控制在必要范围内。
能在 shared 实现的逻辑，不要下沉到平台层。

### 9.3 命名
命名要直接、语义明确。
避免模糊缩写。

### 9.4 注释与文档
所有公开接口和关键模块应具备基础说明。
复杂同步逻辑必须补充注释或文档。

---

## 10. 测试规则

必须为核心逻辑编写测试，优先覆盖：
- 仓储层
- SQLDelight 查询行为
- 同步逻辑
- 冲突处理
- AI 上下文组装

至少需要验证：
- 新增
- 更新
- 软删除
- 增量同步
- 重复同步幂等性
- 基本冲突场景

---

## 11. 禁止事项

Codex 不应做以下事情：

- 不要直接把所有逻辑塞进 UI 层
- 不要让服务端替代本地数据库
- 不要在没有抽象层的情况下直接耦合 AI SDK
- 不要使用数据库文件级同步替代实体级同步
- 不要默认使用复杂 CRDT 方案
- 不要在未说明的情况下引入重量级框架
- 不要破坏已有模块编译
- 不要修改 schema 却不写 migration
- 不要物理删除可同步数据

---

Codex 每次完成任务后，必须输出以下内容：

1. 本次修改了什么
2. 涉及哪些文件
3. 是否包含 schema 或 API 变化
4. 是否新增 migration
5. 当前假设与取舍
6. 下一步最推荐做什么

## 13. 实现优先级
如果没有额外说明，默认按以下顺序推进：

1. 架构与模块边界
2. 本地数据库 schema
3. 仓储层
4. 同步骨架
5. 服务端同步 MVP
6. AI 抽象层
7. UI 接入
8. 高级功能（搜索、总结、备份、推送等）

## 14. 最终目标
该项目的目标不是做一个“在线 CRUD 应用”，而是做一个：

**本地优先、跨设备同步、可扩展 AI 能力、隐私友好的个人日记系统。**
